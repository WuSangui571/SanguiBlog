# Backend Directory Structure

> SanguiBlog backend is a Spring Boot 3.5 / Java 21 REST API. Document real project structure here and follow it before adding code.

---

## Scope / Trigger

Read this spec before changing anything under `SanguiBlog-server/`, especially when adding controllers, services, entities, DTOs, security filters, uploads, AI features, analytics, sitemap, or system-monitor behavior.

---

## Root Layout

```text
SanguiBlog-server/
|-- pom.xml
|-- src/main/java/com/sangui/sanguiblog/
|   |-- SanguiBlogServerApplication.java
|   |-- bootstrap/
|   |-- config/
|   |-- controller/
|   |-- exception/
|   |-- model/
|   |   |-- dto/
|   |   |-- entity/
|   |   |-- permission/
|   |   +-- repository/
|   |-- security/
|   |   +-- botguard/
|   |-- service/
|   |   |-- ai/
|   |   |   +-- rag/
|   |   +-- ...
|   +-- util/
|-- src/main/resources/
|   |-- application.yaml
|   +-- application-dev.yaml
+-- src/test/java/com/sangui/sanguiblog/
```

Database schema is not generated by Hibernate. The canonical schema is `sanguiblog_db.sql`, and `spring.jpa.hibernate.ddl-auto` is `none`.

## Layer Contracts

| Layer | Location | Responsibility | Do Not Put Here |
|-------|----------|----------------|-----------------|
| Controller | `controller/*Controller.java` | HTTP mapping, request params/body binding, principal extraction, `ApiResponse` wrapping | Business rules, SQL, file path manipulation beyond request-level normalization |
| Service | `service/*Service.java` | Business rules, transactions, DTO mapping, cross-repository orchestration | HTTP annotations except where framework integration requires it |
| Repository | `model/repository/*Repository.java` | Spring Data JPA interfaces, derived queries, JPQL/native query contracts | Business decisions, request parsing |
| Entity | `model/entity/*.java` | JPA table/column mapping | API-only formatting or frontend-specific fields |
| DTO | `model/dto/*.java` | Request/response payloads | Persistence behavior |
| Config | `config/*.java` | Security, storage, AI vector store, OpenAPI, web config | Feature business rules |
| Security | `security/**` | JWT, user principal, authentication filters, bot guard | Domain services unrelated to auth/risk |
| Exception | `exception/**` | Project exceptions and `GlobalExceptionHandler` | Per-feature fallback logic |
| Util | `util/**` | Stateless helpers such as IP/referrer parsing | Stateful services |

## Naming Conventions

- Controllers end with `Controller`, use class-level `@RequestMapping`, and return `ApiResponse<T>` unless the endpoint is intentionally non-JSON such as `SitemapController`.
- Services end with `Service`; external capability providers can use a more precise noun such as `SystemMonitorMetricsProvider`.
- Repositories end with `Repository` and live under `model.repository`.
- Entities are singular nouns and map to snake_case tables with `@Table(name = "...")`.
- DTOs end with `Dto`, request objects end with `Request`, and paginated responses use `PageResponse<T>`.
- Admin-only HTTP controllers generally use `Admin*Controller`, while admin-only DTOs use `Admin*Dto`.
- AI chat services are grouped under `service.ai`; RAG/vector-store support is under `service.ai.rag`.

## Controller Pattern

Controllers should be thin and explicit.

```java
@RestController
@RequestMapping("/api/posts")
@RequiredArgsConstructor
public class PostController {
    private final PostService postService;

    @GetMapping
    public ApiResponse<PageResponse<PostSummaryDto>> list(
            @RequestParam(required = false) Integer page,
            @RequestParam(required = false) Integer size,
            @RequestParam(required = false) Long categoryId,
            @RequestParam(required = false) Long tagId,
            @RequestParam(required = false) String keyword) {
        return ApiResponse.ok(postService.listPublished(page, size, categoryId, tagId, keyword));
    }
}
```

Use `@AuthenticationPrincipal UserPrincipal principal` at the controller boundary when a service needs the current user id. Convert it to `Long uid` there; do not pass Spring Security objects deep into generic services unless the service is security-specific.

## Service Pattern

Service methods own transaction boundaries and DTO assembly.

```java
@Service
@RequiredArgsConstructor
public class RegistrationInviteService {
    private final RegistrationInviteRepository registrationInviteRepository;

    @Transactional(readOnly = true)
    public PublicRegistrationInviteVerifyDto verify(String inviteCode) {
        // validate input, load entity, return DTO
    }

    @Transactional
    public AdminRegistrationInviteDto create(Long createdByUserId) {
        // mutate state and save
    }
}
```

Use `@Transactional(readOnly = true)` for read-only service methods. Use `@Transactional` for methods that save entities, update visibility flags, increment counters, write audit rows, sync RAG metadata, or mark sitemap state dirty.

## Feature Placement

| Feature | Existing Entry Points | Extension Rule |
|---------|-----------------------|----------------|
| Posts and archives | `PostController`, `AdminPostController`, `PostService`, `PostRepository` | Extend these paths before creating new post services/controllers |
| Uploads | `UploadController`, `PostAssetService`, `AvatarStorageService`, `StoragePathResolver` | Keep upload validation and storage path normalization centralized |
| Site meta/home settings | `SiteController`, `SiteService`, `HomeBackgroundAdminService`, `site_settings`, `home_background_images` | `/api/site/meta` is the public frontend source of truth |
| AI assistant | `AiChatController`, `AiChatService`, `service.ai.*`, `service.ai.rag.*` | Do not create a second chat pipeline; extend the existing context/advice services |
| Permissions | `PermissionService`, `PermissionDefinition`, `SecurityConfig`, `@PreAuthorize` | Keep `SUPER_ADMIN` fallback where existing upload/admin contracts require it |
| Sitemap/robots | `SitemapController`, `SitemapService` | Keep XML/text responses outside `ApiResponse` |
| System monitor | `AdminSystemMonitorController`, `SystemMonitorService`, `SystemMonitorMetricsProvider` | Extend `SystemMonitorDto` and service path; do not add platform-specific controllers |

## Anti-Patterns

### Do Not Add a Parallel Feature Entry

Wrong:

```text
NewAiChatController + NewAiChatService
```

Correct:

```text
AiChatController -> AiChatService -> existing advice/RAG/session services
```

Why: `.ai` history shows multiple AI assistant regressions came from subtle cross-layer behavior. A second chat pipeline would bypass guest limits, history visibility, current-page context, referenced-post context, system facts, and SSE completion semantics.

### Do Not Put SQL Schema Changes Only in Entity Classes

Wrong: add an entity field and rely on Hibernate to create it.

Correct: update entity, DTO/repository/service contracts, and `sanguiblog_db.sql` together.

Why: `application.yaml` sets `ddl-auto: none`; deployments require explicit SQL.

### Do Not Scatter Static File Path Rules

Wrong: each upload feature manually concatenates OS paths.

Correct: use `StoragePathResolver`, `PostAssetService`, and existing upload folder normalization patterns.

## Good / Base / Bad Cases

| Case | Expected Implementation |
|------|-------------------------|
| Good | Add a new admin post list filter by extending `AdminPostController`, `PostService.adminList`, repository criteria, frontend `api.js`, and existing admin view state |
| Base | Add a new read-only public endpoint in an existing controller, wrapping data with `ApiResponse.ok(data)` |
| Bad | Create a new controller/service pair for the same business object because the existing one is large |

## Tests Required

- Backend service changes: add or update JUnit tests under matching package in `SanguiBlog-server/src/test/java/com/sangui/sanguiblog/...`.
- Controller/security boundary changes: add tests for authorization and response status, using existing controller/security tests as examples.
- Cross-layer API changes: update frontend `api.js` and any static tests that assert string/structure contracts.
